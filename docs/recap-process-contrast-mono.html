What moves the finish line?

The same two tasks - task 1 takes a fix and a re-review, task 2 passes first time - run three ways, on one clock, with one reviewer. Watch what actually moves the finish, and what doesn't.

          1       2       3       4       5       6
          ── wall time ─────────────────────────▶

STOP & WAIT · they take turns · 6 cycles

agent     [task 1]· idle ·[fix 1 ]· idle ·[task 2]· idle ·│
reviewer          [rev 1 ]        [rev f1]        [rev 2 ]│
                                                     done ┘

ASYNC · one agent, never waits · 4 cycles

agent     [task 1][task 2][fix 1 ]        │
reviewer          [rev 1 ][rev 2 ][rev f1]│
                                     done ┘

WORKTREES · parallel build, then merge back · 5 cycles

agent     [t1, t2]· idle ·[fix 1 ]· idle ·[merge ]│
reviewer          [rev 1 ][rev 2 ][rev f1]        │
                                             done ┘

Six cycles, then four, then five. Stop-and-wait to async is a third faster - and when async finishes, stop-and-wait still has a whole review to go. Async to worktrees gives a cycle back. Parallel building collapses the agent's time, but the reviewer reads the same things in the same order, so review stays the bottleneck. More agents just buys more idle agents and a merge.

And that's the optimistic read. Off the clock, worktrees add cost the diagram can't time:

  • drift - trees diverge from main and from each other while they run
  • merge conflicts - reconciling them is serial, error-prone work, and can force a re-review
  • cross-tree dependencies - when task 2 leans on task 1, you hold the coupling in your own head
